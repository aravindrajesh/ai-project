<h1>Weasel Information</h1>

<p>In 1986, Richard Dawkins published his book, "The Blind Watchmaker". In the book, he
utilized programs for computers to illustrate some issues in evolutionary biology. One
of these programs has come to be called "weasel" because of the use in it of a short
snippet of Shakespeare, "METHINKS IT IS LIKE A WEASEL".</p>

<p>The "weasel" program is aimed at illustrating the difference between what Dawkins
called "single-step selection", or random search, with a process of "cumulative
selection", that employs something with several components of <i>natural selection</i>.</p>

<p>The demonstration shows that cumulative selection, premised on natural selection,
is far more efficient than random search on the problem posed, that of finding a
particular string, that snippet from Shakespeare. It has been a
favorite target of religious antievolutionist rhetoric ever since.</p>

<p>Dawkins' description of the "weasel" program can be summarized briefly:<p>
<ol>
<li>Use a set of characters that includes the upper case alphabet and a space.
<li>Initialize a population of n 28-character strings with random assignments of characters from our character set.
<li>Identify the string closest to the target string in the population.
<li>If a string matches the target, terminate.
<li>Base a new generation population of size n upon copies of the closest matching string, where each position has a chance of randomly mutating, based upon a set mutation rate.
<li>Go to step 3.
</ol>

gdwr.run_onestep = function() {

// gdwr.console.value += gdwr.gen + " (1)\n";

// Show generation in console
// this.console.value += "Gen. " + this.gen + "\n";

// gdwr.console.value += gdwr.gen + " (2)\n";

gdwr.thismatch++;

if (gdwr.thismatch > gdwr.maxmatch) {

} else if (gdwr.thismatch == gdwr.maxmatch) {
//ta.value += "Gen. " + gen + ", " + maxmatch + " letters, " + makestr(pop[gdwr.maxndx]) + "\n";

// ta.value += gen + " " + ii + " " + thismatch + " " + makestr(pop[ii]) + "\n";

gdwr.pop[ii][jj] = gdwr.pop[ii][jj].toLowerCase();
}

gdwr.exactcopycnt++;
}

// A utility function that returns true if a string contains only
// whitespace characters
function isblank(s) {
for(var i = 0; i < s.length; i++) {
var c = s.charAt(i);
if ((c != ' ') && (c != '\n') && (c != '')) return false;
}
return true;
}

// This is the function that performs form verification. It is invoked
// from the onsubmit event handler. The handler should return whatever
// value this function returns.
function verify(f) {
var msg;
var empty_fields = "";
var errors = "";

// Loop through the elements of the form, looking for all Text and
// Textarea elements that don't have an "optional" property defined.
// Then check for fields that are empty and make a list of them. Also, if
// any of these elements have a "min" or a "max" property defined, verify
// that they are numbers and are in the right range. If the element has a
// "numeric" property defined, verify that it is a number, but don't check
// its range. Put together error messages for fields that are wrong.
for(var i = 0; i < f.length; i++) {
var e = f.elements[i];
if (((e.type == "text") || (e.type == "textarea")) && !e.optional) {
// First check if the field is empty
if ((e.value == null) || (e.value == "") || isblank(e.value)) {
empty_fields += "\n " + e.name;
continue;
}

// Now check for fields that are supposed to be numeric
if (e.numeric || (e.min != null) || (e.max != null)) {
var v = parseFloat(e.value);
if (isNaN(v) ||
((e.min != null) && (v < e.min)) ||
((e.max != null) && (v > e.max))) {
errors += "- The field " + e.name + " must be a number";
if (e.min != null)
errors += " that is greater than " + e.min;
if (e.max != null && e.min != null)
errors += " and less than " + e.max;
else if (e.max != null)
errors += " that is less than " + e.max;
errors += ".\n";
}
}
}
}

// Now, if there were any errors, display the messages, and
// return false to prevent the form from being submitted.
// Otherwise, return true.
if (!empty_fields && !errors) return true;

msg = "_____________________________________________________ _\n\n"
msg += "The form was not submitted because of the following error(s).\n";
msg += "Please correct these error(s) and re-submit.\n";
msg += "_____________________________________________________ _\n\n"

if (empty_fields) {
msg += "- The following required field(s) are empty:"
+ empty_fields + "\n";
if (errors) msg += "\n";
}
msg += errors;
alert(msg);
return false;
}

<h3>Try 1</h3>

<p>A common antievolutionist misconception (which, when repeated after due notice becomes a <i>lie</i>) is that Dawkins stated or
implied that letters, once matched, were "locked" or "frozen" somehow, and could not be subject to mutation in further generations.
This is simply poppycock. This implementation of "weasel" not only does not "lock" correct letters, it even provides feedback to
the user on the issue of a later generation having a <i>fewer</i> letters matched by its closest candidate string to the
target phrase.</p>

<p>I have seen it argued that since Dawkins' short listing of some example strings during a run did not <i>show</i> such an
instance of fewer letters matching, that it must be a rule used in the program. This is just sloppy thinking, a case where
someone is overeager to infer a general principle from a particular observation. As such, it may be useful as a warning about
the problem of induction.</p>

<p>One can work out analytically the conditions needed in order to observe a subsequent generation in a "weasel" run that has
a best candidate string with fewer matches to the target string. I'll try to work up some equations that express the probabilities
in that way, but in the meantime I can just describe the circumstances that apply and you can use the simulation above to
explore things.</p>

<p>In making a new generation, every character of every new string is tested to see if it gets mutated from what the closest-matching
string in the previous generation had. So the mutation rate combined with the length of the target string gives us an expectation
for the average number of characters that will be altered in each new string in the new generation. For the standard phrase, there
are 28 characters, so a mutation rate of about 3.57% yields an expectation of one changed character per new string. There will be
a characteristic expectation of how often a completely unchanged string will be seen in the new generation. If such a string that
is the same as the parent string from the previous generation exists, the new generation will have at least as good a match to the
target as the previous generation.</p>

<p>So, for a new generation to perform worse than a previous generation, we expect that to happen when the odds of having at least
one string in the new population that is the same as the parent in the previous generation is low, <i>and</i> when the odds of
changing an already-matched character in each of the strings in the new generation is relatively high. This set of conjoint
conditions applies when there is a relatively high mutation rate coupled with a small population size. For example, if one sets
a mutation rate of 7.3%, one expects about two characters in each new string to be altered. If population size is set to 10,
then one will see the step-back counter start rising, usually once over 12 characters match to the target.</p>

<p>What about the printed set of outputs in Dawkins' "The Blind Watchmaker"? If Dawkins used a reasonable population size of
between 100 and 1000 strings, it is difficult to come up with a mutation rate that simultaneously permits occasional convergence on the
target string and shows the stepping-back behavior. With reasonable mutation rates and population sizes, the odds of observing
the stepping-back of performance between successive generations becomes very small. This property shows that all the bloviation
about "locking" of characters is simply a persistent creationist myth.</p>

<!--<h3>Try 2</h3> -->

<p>The "weasel" program as described by Richard Dawkins obviously
works. If random search were used, we would not expect the target
string to be found in our lifetimes, or the lifetime of our
species. Antievolutionists have made a false charge that the "weasel"
program can only obtain its better performance by "smuggling"
information into the result. This has recently (2007) been expressed
in an essay by William Dembski and Robert Marks, where they
erroneously claim that Dawkins was using a "partitioned search"
procedure. This differs from what Dawkins actually described and
implemented by protecting individual characters in strings from
mutation during copying based on matching the corresponding character
in the target. In this alternate version described by Dembski and
Marks, individual characters are "locked" once they match to the
target string, and they claim that it is this additional constraint
that permits any "weasel" program -- and even any evolutionary
computation at all -- to out-perform random search. Note this
egregious mishandling of Dawkins' description from
their <a href="http://marksmannet.com/EILab/WeaselWare.html">website</a>:</p>


<blockquote>
The active information supplied by the "divide and conquer" oracle is necessary to perform the search. In essence, Dr. Dawkins concurs when he writes

<p><tt>
"Chance is a minor ingredient in the Darwinian recipe [as exemplified in the WEASEL example], but the most important ingredient is the cumulative selection [i.e. the divide and conquer oracle] which is quintessentially nonrandom." (Italics not added ) [5].
</tt></p>

In evolutionary computing, the active information can be generated by a programmer skillfully querying an information rich oracle.
</blockquote>

<p>What needs to be understood about this is that the "partitioned
search" analyzed by Dembski and Marks has nothing to do with the
program described by Richard Dawkins. Dawkins did not state that
individual letters could be protected, and in fact has stated that
doing so would be bad didactically, since that would be at variance
with how natural selection works in biology (2000, personal
communication). As the program implemented here shows, a "weasel"
that works without benefit of the "partitioned search" of Dembski
and Marks nonetheless performs far better than random search,
contrary to the claims Dembski and Marks have made.</p>
